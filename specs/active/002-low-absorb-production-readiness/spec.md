## 1. 需求背景

PR #3 已完成 Low Absorb 主流程基线，包括数据源基础接入、AI 成本链、信号排序、交易工作台、情绪许可、回测示例、复盘报告、持久化和端到端测试。下一阶段的重点不是继续扩展页面功能，而是把系统从“本地可演示、可验证”推进到“真实数据可接入、异常可降级、外部联调可控制、本机可长期运行”的生产化状态。

本阶段只覆盖非 UI 生产化主线：

1. 真实行情数据源稳定性；
2. 英伟达 AI 服务器成本链半自动采集与人工审核；
3. 日线级真实回测引擎接入；
4. 飞书全类型通知受控真实联调；
5. Windows 本机长期运行部署与生产配置。

页面中文化、首页/菜单/按钮文字、视觉体验和整体 UI 优化不进入本阶段，后续单独立项。

本阶段仍遵守 Low Absorb 的人工执行定位：系统只生成研究信号、人工交易计划、风控提醒、飞书通知、人工成交记录和复盘材料；用户始终在国内券商 App 中手动完成真实交易。

## 2. 目标

- 将行情、成本链、回测、通知和生产配置从“fixture 或本地示例可用”提升为“真实数据可接入、异常可解释、结果可审计、本机可长期运行”。
- 建立统一的数据质量模型，覆盖来源、采集时间、新鲜度、字段完整性、fallback、冲突和 fail-closed 原因。
- 让成本链资料进入“半自动采集、人工审核、生效版本、回滚审计”的受控流程。
- 接入日线级真实回测，输出可复现、可解释、明确限制的研究结果。
- 完成飞书通知真实联调能力，但默认不产生外部副作用。
- 明确 Windows 本机长期运行的环境变量、存储路径、健康检查、日志、备份、恢复和回滚要求。

## 3. 非目标

- 不做 UI 中文化全局复查；该工作后续另开 `003-ui-localization-and-polish` 或等价 UI spec。
- 不做视觉设计重构、页面布局重构或首页体验优化。
- 不接入券商交易执行 API。
- 不实现真实下单、自动买卖、自动委托或一键交易。
- 不新增独立按钮文案：`买入`、`卖出`、`下单`、`自动交易`。
- 不承诺收益，不把回测结果展示为确定收益。
- 不在前端直接调用行情源、成本链资料源或研报源。
- 不用 AI 生成行情数据、财务数据、成本权重或历史回测数据。
- 不在未获授权时新增 Python 或前端依赖。

## 4. 用户故事

1. 作为交易研究用户，我希望系统能稳定读取真实 A 股、全球行情和美股风险输入，以便在数据缺失、过期或冲突时停止生成误导性交易建议。
2. 作为策略维护者，我希望英伟达 AI 服务器成本链能半自动采集、人工审核、版本化生效，以便信号排序依据可追溯、可回滚。
3. 作为策略研究者，我希望回测引擎使用真实日线历史数据和固定策略参数运行，以便验证策略在不同市场阶段下的稳健性。
4. 作为系统运维者，我希望飞书通知能在受控环境中真实联调，以便上线前确认推荐卡片、风控提醒、成交回填提醒和复盘报告可以可靠送达。
5. 作为本机部署负责人，我希望有清晰的环境变量、存储、日志、健康检查和备份恢复方案，以便系统可在 Windows 本机长期运行。

## 5. 功能需求

### FR-1：真实行情数据源稳定性

- 后端必须继续通过 `agent/src/low_absorb/` 下的数据源 provider 访问外部数据，前端不得直接访问外部行情源。
- Claude Code 实现阶段可以参考 `a-stock-data` skill 的 A 股数据源方案和 Python 调用范式，但必须封装在 Low Absorb 后端 provider 内，不得把 skill 示例代码散落到前端、页面或策略调用点。
- A 股行情必须采用多源备份策略，不能依赖单一接口：
  - 一级优先：`mootdx` 通达信 TCP 行情，用于 K 线、盘口、逐笔、财务快照、F10 等相对稳定数据；
  - 二级优先：腾讯财经 HTTP 接口，用于实时价格、涨跌幅、PE/PB、市值、换手率、指数和 ETF 等补充数据；
  - 三级备份：百度股市通、新浪财经、巨潮、同花顺等公开源，用于 K 线、财报、公告、热点或题材补充；
  - 东方财富仅用于其独有数据或必要备份，如行业板块、资金流、龙虎榜、解禁、研报、个股新闻，并必须串行限流、复用会话、记录风控状态。
- 全球行情和美股行情必须作为市场情绪输入，支持全球风险偏好、主要科技股风险状态和异常降级。
- provider 必须定义数据源优先级、单源失败重试、跨源 fallback、熔断冷却、健康评分和冲突容忍阈值。
- 每次数据获取必须记录实际来源、市场日期、采集时间、数据新鲜度、字段完整性、失败原因、fallback 过程和冲突字段。
- 同一关键字段存在多个来源时，必须记录实际采用来源；当多源数据冲突超过配置容忍阈值时，应降级为观察或拦截，并记录冲突原因。
- 数据缺失、字段异常、接口失败、返回空数据、超过新鲜度阈值或全部来源失败时必须 fail-closed，不得静默生成新的交易建议。
- provider 可以支持本地/dev fixture fallback，但 fallback 必须显式标记；生产模式默认不得用 fixture 伪装真实数据。
- 生产模式下多源全部失败时，只能返回失败状态和拦截原因。
- 数据源状态必须能被 API 和诊断入口读取。
- 如果 `a-stock-data` 方案或全球行情方案需要当前项目尚未安装的 Python 依赖，Claude Code 必须停止并请求 Codex 更新契约和用户授权，不得自行修改依赖配置或安装依赖。

### FR-2：英伟达 AI 服务器成本链半自动采集与人工审核

- 成本链必须维护版本化数据，包括 `GB200 NVL72`、`GB300 NVL72`、`custom/manual` 和后续可扩展版本。
- 成本链条目必须包含组件、成本权重或权重区间、代际变化、相关板块、A 股映射、主板可交易映射、信号权重、来源、来源类型、置信度、as_of、估算说明。
- 第一阶段采用“半自动采集审核”：后端任务或脚本可采集公开资料并生成候选版本，但不得自动生效。
- 自动采集结果必须进入待审核状态；低置信度或来源不充分的数据不得自动影响交易计划排序。
- 人工审核通过后，候选版本才能成为有效成本链版本并被信号排序使用。
- `custom/manual` 必须支持人工修订、保存、回滚和版本记录。
- 所有成本链更新必须可审计：记录更新时间、来源、变更前后差异、审核状态和操作者或任务来源。
- 成本链更新失败时必须保留上一有效版本，不得污染当前生效版本。

### FR-3：日线级真实回测引擎接入

- 第一阶段回测只支持日线级真实历史数据，不实现分钟级 14:45 尾盘触发、次日 09:30-10:00 抗噪和 10:00 监督回测。
- 回测引擎必须使用后端历史行情数据、固定策略参数、成本链版本、市场情绪规则和人工执行假设。
- 回测输入必须包含数据版本、参数快照、股票池规则、交易日期范围、过滤漏斗和风险预算规则。
- 回测输出必须包含信号漏斗、交易计划数量、触发/失效原因、胜率、平均 R、最大回撤、样本数、分支归因、敏感性分析和异常样本说明。
- 回测必须区分“系统建议”和“人工成交假设”，不得把回测信号等同真实成交。
- 回测结果必须明确标注数据来源、样本范围、参数版本和限制，不得展示收益承诺式文案。
- 回测任务必须可重复运行，同一输入配置应能得到可解释的一致结果。
- 回测历史数据缺失时，任务必须失败并提示缺失区间和来源，不得生成虚假结果。

### FR-4：飞书全类型通知受控真实联调

- 飞书 Webhook 必须通过环境变量或受控设置配置，前端和 API 响应不得暴露完整 Webhook。
- 真实发送必须默认关闭，只有显式配置或环境变量允许时才能执行真实 Webhook 调用。
- 通知类型必须覆盖推荐卡片、10:00 风控提醒、人工成交回填提醒、收盘复盘报告和测试卡片。
- 飞书通知必须具备幂等键，避免重复推送同一计划、同一风险提醒、同一成交提醒或同一报告。
- 默认测试必须走 mock，不调用真实飞书。
- 真实联调测试必须受环境变量保护；缺少环境变量时必须跳过，不得失败或真实发送。
- 通知结果必须持久化：记录通知类型、目标对象、幂等键、真实发送标记、发送状态、错误原因和发送时间。
- Webhook、Token、Key 不得明文出现在前端、日志、测试快照或 API 响应中。

### FR-5：Windows 本机长期运行部署与生产配置

- 第一阶段生产化部署目标为 Windows 本机长期运行。
- 必须定义本地/dev、测试、生产三类配置边界。
- 必须定义生产环境变量清单，包括行情数据源配置、飞书 Webhook、存储路径、日志级别、fixture fallback 开关、数据新鲜度阈值和真实发送开关。
- 必须明确持久化存储方案、备份位置、状态文件或数据库迁移策略。
- 必须提供健康检查 API 或等价诊断入口，覆盖后端服务、数据源状态、存储状态、成本链有效版本和飞书配置状态。
- 必须提供启动、停止、日志查看、备份、恢复和回滚说明。
- 生产配置不得硬编码密钥、Webhook、真实账号、个人路径或本地机器专属路径。
- 生产 readiness 必须检查 fixture fallback 是否误开启、真实数据源是否可用、存储是否可读写、成本链是否存在有效版本。

## 6. 非功能需求

- **NFR-1 稳定性**：外部数据源异常不得导致服务崩溃；应返回可解释错误和安全状态。
- **NFR-2 可审计性**：行情、成本链、回测、通知、设置变更必须能追踪来源和时间。
- **NFR-3 安全性**：密钥、Webhook、Token 不得明文出现在前端、日志、测试快照或 API 响应中。
- **NFR-4 可测试性**：真实外部联调必须被环境变量保护；默认测试使用 mock/fixture。
- **NFR-5 可维护性**：新增 provider、调度器、回测引擎和部署配置必须遵循 Low Absorb 模块边界。
- **NFR-6 性能**：回测和数据更新任务应避免阻塞普通页面请求；长耗时任务应提供状态查询或异步执行方案。

## 8. 边界条件

| 场景 | 预期行为 |
|------|----------|
| A 股主数据源不可用但备份源可用 | 自动切换到备份源，响应中标记实际来源和 fallback 原因 |
| A 股多个来源数据冲突 | 超出容忍阈值时降级为观察或拦截，并记录冲突字段、来源和值 |
| A 股所有来源均不可用 | 不生成信号或交易计划，诊断入口展示全部来源失败原因 |
| 全球行情源不可用 | 市场情绪返回观察或拦截，不返回允许 |
| 数据超过新鲜度阈值 | 策略扫描 fail-closed，返回数据过期原因 |
| 成本链来源低置信度 | 可展示为研究信息，但不得自动提高交易优先级 |
| 成本链更新失败 | 保留上一有效版本，记录失败原因 |
| 成本链候选未审核 | 不影响当前有效排序 |
| 回测历史数据缺失 | 回测任务失败并提示缺失区间，不生成虚假结果 |
| 飞书 Webhook 未配置 | 通知返回失败原因，不抛出未处理异常 |
| 飞书真实联调未开启 | 测试卡片只走 mock，不调用真实 Webhook |
| 生产配置缺少必填环境变量 | readiness 明确报错，不使用不安全默认值 |
| 生产模式误开启 fixture fallback | readiness 失败并提示配置风险 |

## 9. 金融安全边界

本阶段继续明确：Vibe-Trading 是金融研究和人工执行辅助系统，不是自动交易系统。

禁止：

- 真实券商下单；
- 自动买卖；
- 自动委托；
- 接入券商交易执行 API；
- 一键交易；
- 独立执行按钮文案：`买入`、`卖出`、`下单`、`自动交易`。

允许：

- 生成研究信号；
- 生成交易计划；
- 推送飞书；
- 复制人工下单信息；
- 记录人工成交；
- 标记失效；
- 生成风控提醒；
- 记录卖出；
- 生成复盘报告。

## 10. 风险等级说明

本阶段为 A 类高风险任务，原因：

- 涉及成本链资料采集、审核和信号排序依据；
- 涉及真实回测引擎和用户可能据此决策；
- 涉及真实飞书 Webhook 联调；
- 涉及本机生产部署配置、密钥管理、备份恢复和健康检查。

## 11. 相关宪法约束

- 后端使用 Python、FastAPI，外部数据源必须通过后端封装。
- 前端使用 React + TypeScript + Vite，前端不包含策略规则。
- 禁止真实券商下单、自动买卖、自动委托和禁用按钮文案。
- 禁止硬编码密钥、Webhook、Token 或真实账户信息。
- 错误必须显式处理，禁止静默吞掉异常。
- 后端公共 API 必须有类型注解。
- 后端 API 响应格式统一为 `{ success: bool, data: any, error: string | null }`。
- 测试优先，后端使用 pytest，前端使用 vitest + React Testing Library。
