# Low Absorb 后续优化技术方案

## 1. 方案定位

本方案服务于 `spec.md` 中定义的 Low Absorb 后续优化需求。目标是把 AI 主板低吸模块从示例驱动的页面升级为具备真实数据源、产业链权重、人工执行闭环和复盘能力的决策型工作台。

本模块仍然是人工执行交易 Copilot，不是券商执行系统。后端可以生成信号、交易计划、风险提醒、飞书消息和复盘报告；前端只能展示建议、解释、人工记录入口和风险状态。

## 2. 关键约束复述

1. 不实现真实券商下单。
2. 不接入券商交易 API。
3. 不实现自动买卖。
4. 不出现独立按钮文案：买入、卖出、下单、自动交易。
5. 允许动作文案仅限人工执行语义，例如推送飞书、复制人工下单信息、记录人工成交、标记失效、生成风控提醒、记录卖出。
6. 前端不得包含策略规则。
7. 策略扫描、阈值判断、数据新鲜度检查、成本链权重和信号排序必须在后端完成。
8. 前端不得硬编码真实英伟达成本权重。
9. 行情数据必须由 Python 后端数据源模块获取，不由 AI 生成。
10. 成本链数据必须包含来源、来源类型、置信度和 as_of 日期。

## 3. 架构决策

### 3.1 后端分层

后端继续使用 `agent/src/low_absorb/` 作为 Low Absorb 模块根目录。

建议分层：

- 数据源层：封装 A 股、全球行情、美股行情、新闻、行业和 F10 数据。
- 策略层：执行宏观情绪门控、AI 产业链门控、主板过滤、技术过滤和排序。
- 产业链层：维护成本链模型、板块强度、股票角色和主板映射。
- 人工执行层：生成交易计划、飞书建议、人工成交回填、持仓风险监督。
- 报告层：生成收盘报告、风险复盘和历史归档。
- API 层：向前端提供 snapshot、scan-tail、chain、sentiment、backtest、reports、settings 等接口。

### 3.2 前端分层

前端继续使用 Low Absorb 页面组：

- `/low-absorb`：交易工作台。
- `/low-absorb/sentiment`：市场情绪。
- `/low-absorb/chain`：AI 产业链。
- `/low-absorb/backtest`：策略回测。
- `/low-absorb/reports`：复盘报告。
- `/low-absorb/settings`：系统设置。

前端职责：

- 展示后端返回的信号、计划、情绪许可、成本链、持仓风险和报告。
- 发起人工动作请求，例如推送飞书、记录人工成交、标记失效。
- 展示数据源状态、失败原因和人工可解释文本。

前端不得：

- 计算策略入选条件。
- 计算真实成本链权重。
- 直接访问外部行情源。
- 生成或执行券商订单。

## 4. 数据源方案

### 4.1 A 股数据源

第一阶段优先使用 `a-stock-data` 能力补充 A 股数据。

目标数据：

- 日线行情。
- 分钟线行情。
- 市场宽度。
- 行业或概念强度。
- 个股资金流。
- 新闻事件。
- F10 基础资料。

数据源必须通过统一数据提供者契约接入。

建议契约方法：

- `get_market_breadth(trade_date, at)`
- `get_daily_bars(symbols, end, lookback)`
- `get_intraday_bars(symbol, trade_date, interval)`
- `get_industry_return(industry, trade_date, start, end)`
- `get_chain_branch_strength(trade_date, lookback)`
- `get_stock_news(symbol, trade_date)`
- `get_stock_f10(symbol)`

### 4.2 全球行情与美股行情

建议第一阶段采用可替代公开数据源：

- 美股行情：`yfinance` 或 `Stooq`。
- 全球指数：Yahoo Finance、Stooq、FRED 或等价公开源。

数据用途：

- 全球风险偏好。
- 海外科技股风险状态。
- AI 服务器产业链全球映射。
- 市场情绪交易许可判断。

### 4.3 数据质量策略

数据质量必须包含：

- 数据源标识。
- 拉取时间。
- 市场日期。
- 数据新鲜度。
- 缺失字段检查。
- 异常降级原因。

策略扫描使用 fail-closed：

- 数据缺失时不生成交易建议。
- 数据过期时不生成交易建议。
- 外部源异常时返回拦截原因。

## 5. 英伟达 AI 服务器成本链模型

### 5.1 版本模型

成本链使用版本化模型：

- `GB200 NVL72`
- `GB300 NVL72`
- `custom/manual`

`GB200 NVL72` 和 `GB300 NVL72` 作为内置版本。`custom/manual` 作为用户可编辑版本。

### 5.2 成本链字段

成本链组件字段：

| 字段 | 说明 |
|------|------|
| component | 组件名称 |
| cost_weight | 成本权重 |
| cost_weight_range | 权重区间 |
| cost_increase_vs_previous_generation | 代际成本增幅 |
| related_sector | 相关板块 |
| a_share_leaders | A 股龙头 |
| tradable_mainboard_mapping | 主板可交易映射 |
| signal_weight | 对信号排序的权重 |
| data_source | 数据来源 |
| source_type | 来源类型 |
| confidence | 置信度 |
| as_of | 数据截至日期 |
| methodology_note | 估算方法说明 |

### 5.3 来源类型

来源类型建议：

- 官方资料。
- 券商估算。
- 产业研究。
- 手动维护。

### 5.4 成本链使用原则

- 成本权重不在前端硬编码。
- 真实或估算数据必须可追溯。
- 低置信度数据可以展示，但不得单独决定交易建议。
- 成本链信号权重只影响排序和解释，不替代技术过滤。

## 6. 策略与排序方案

### 6.1 信号漏斗

14:45 扫描漏斗：

1. 股票池。
2. 宏观情绪门控。
3. AI 产业链门控。
4. 主板过滤。
5. 技术过滤。
6. 交易计划生成。

### 6.2 排序因子

信号排序建议使用：

- 技术条件评分。
- AI 分支强度。
- 成本链 signal weight。
- 数据新鲜度。
- 拦截或降级状态。
- 风险预算约束。

### 6.3 降级与拦截

弱分支处理：

- 如果 AI 分支排名靠后且趋势斜率为负，拦截或降级。
- 如果技术条件合格但产业链弱，降低优先级。
- 如果技术条件相近，强成本增量板块提高优先级。

## 9. 存储方案

需要持久化：

- 信号。
- 交易计划。
- 人工成交。
- 人工持仓。
- 持仓风险。
- 飞书通知。
- 成本链设置。
- 数据源设置。

## 11. 风险与降级

| 风险 | 处理 |
|------|------|
| 外部行情源不可用 | fail-closed，不生成建议 |
| 成本链资料存在估算 | 显示置信度和来源类型 |
| 数据源返回字段不稳定 | 数据源层做字段校验和错误归因 |
| 前端误展示策略规则 | 策略判断全部后端化 |
| 用户误解为自动交易 | 页面文案强调人工执行 |

## 12. 待确认事项

1. A 股真实数据源优先级是否固定为 `a-stock-data`，还是允许 Tushare、东方财富、腾讯行情作为备用源。
2. 美股与全球行情是否接受 `yfinance` 作为第一阶段默认源。
3. 英伟达 AI 服务器成本链是否需要区分“官方确认数据”和“券商估算数据”的展示层级。
4. custom/manual 成本链版本是否需要支持多套用户自定义方案。
5. 收盘报告是否需要按交易日自动生成，还是先由用户手动触发生成。
